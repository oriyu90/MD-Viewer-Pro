MD Viewer Pro 設計リスク分析レポート
作成日: 2026-06-30
対象: /Users/yuki/Used_ai/MD Viewer Pro

概要
----
MD Viewer Pro は PySide6 + QWebEngineView で Markdown の閲覧・編集・PDF/HTML 書き出しを行う macOS 向けデスクトップアプリです。
現行実装では、過去に危険だった「Markdown 生HTMLの未サニタイズ」「QtWebEngine sandbox の常時無効化」「外部画像の同期取得」「PDF loadFinished の接続順競合」はかなり改善されています。

一方で、まだ残っている大きなリスクは次の4系統です。

- WebEngine / WebChannel / contenteditable を同じプレビュー面で扱うため、未信頼文書とアプリ内ブリッジの境界が薄い。
- MD編集モードが HTML -> Markdown の独自逆変換に依存しており、保存時に文書を壊しやすい。
- ファイル読み込み・一時ファイル・設定保存に安全弁が少なく、大きなファイルや異常終了に弱い。
- ソース、仮想環境、ビルド成果物、配布物、古い依存記述が混在しており、配布再現性とレビュー性が低い。


重大度: 高
1. MD編集モードで文書が壊れる可能性が高い
該当箇所:
- main.py:694-837 _HTML2MD
- main.py:2239-2242 contenteditable の input から innerHTML を Python 側へ送信
- main.py:2442-2446 _on_md_content_changed()
- main.py:2524-2536 モード切替時の innerHTML 回収

内容:
MD編集モードでは、Markdown 原文ではなく QWebEngine 内の contenteditable HTML を編集し、その innerHTML を独自 HTMLParser で Markdown に戻しています。
HTML -> Markdown は本質的に可逆ではなく、現在の変換器も限定的です。テーブル、ネストしたリスト、コードブロック、リンク内装飾、インライン HTML、空行、属性つき要素などは元の Markdown と同じ形に戻る保証がありません。

起こり得る問題:
- MD編集モードで少し触っただけで、保存時に Markdown の構造や空行が変わる。
- 表・リスト・引用・コードブロックが崩れる。
- 外部画像を data URI に置換した状態や、編集補助用 DOM の混入が保存内容に影響する。
- ユーザーから見ると「閲覧しただけ」「軽く編集しただけ」なのにファイル差分が大きくなる。

改善案:
- Markdown 原文を正とする TXT編集を主編集経路にし、MD編集は限定機能として明示する。
- 保存前に HTML 逆変換後の差分確認を出す。
- HTML -> Markdown 変換を専用ライブラリに寄せ、対応範囲をテスト化する。
- MD編集で扱える構文を限定し、非対応構文を検出したら警告する。


重大度: 高
2. WebChannel が全モードで公開され、未信頼文書とアプリ権限の境界が弱い
該当箇所:
- main.py:682-688 ClipboardBridge.copyText()
- main.py:1470-1475 bridge / clipboard を QWebChannel に登録
- main.py:2231-2255 全モードで webchannel_js を HTML に注入
- main.py:2261-2267 自前 JavaScript と WebChannel を同一ページに注入

内容:
Markdown 本文は _sanitize_html() で script や on* 属性を落としていますが、最終 HTML にはアプリ側の JavaScript と QWebChannel が常に注入されます。
特に clipboard ブリッジは全モードで公開され、JavaScript から QApplication.clipboard().setText() を呼べる設計です。現状では Markdown 由来の script は除去されますが、サニタイザの抜け・将来の機能追加・外部ライブラリ出力の変化があると、未信頼文書がアプリブリッジへ触れる余地ができます。

起こり得る問題:
- サニタイズ漏れが1つ出るだけで、未信頼 Markdown がクリップボードを書き換えられる。
- WebChannel に将来メソッドが増えた場合、影響範囲が一気に広がる。
- 表示用 HTML、編集用 HTML、アプリ操作用 JS の境界が曖昧で、セキュリティレビューが難しい。

改善案:
- WebChannel は必要なモードだけで有効化する。閲覧モードでは clipboard bridge もできれば無効にする。
- コピーは可能なら QWebEngine 標準機能やユーザー操作に閉じる。
- ブリッジメソッドは入力サイズ制限・呼び出し元状態チェックを入れる。
- 表示用ページと編集用ページを論理的に分離し、未信頼本文がアプリ API に触れない構成にする。


重大度: 高
3. ファイル読み込みにサイズ上限・バイナリ判定がない
該当箇所:
- main.py:2907-2910 起動時ファイル選択で All Files を許可
- main.py:2938-2954 _load_file()
- main.py:2975-2978 通常ファイル選択で All Files を許可

内容:
任意ファイルを開けるうえ、読み込み時にサイズ上限やバイナリ判定がありません。さらに latin-1 fallback はほぼ任意のバイト列を文字列化できるため、大きなバイナリでも「テキスト」として読み込まれます。

起こり得る問題:
- 巨大ファイルやバイナリを開いて UI が固まる、メモリを大量消費する。
- 誤って非テキストファイルを開き、保存し直して破壊する。
- Markdown 変換・HTML生成・WebEngineロードが巨大入力で連鎖的に重くなる。

改善案:
- 開く前にファイルサイズ上限を確認する。
- 先頭数KBを読んで NUL バイトや制御文字比率でバイナリ判定する。
- latin-1 fallback は確認ダイアログ付きにする。
- All Files は残す場合でも、非 .md/.txt は警告する。


重大度: 中
4. HTML サニタイザの allowlist が広く、style / class / id を許可している
該当箇所:
- main.py:846-878 許可タグ・属性
- main.py:913-918 style 属性の簡易検査
- main.py:982-986 _sanitize_html()

内容:
script や iframe などは落としていますが、グローバル属性として class, id, style, align を許可しています。style は "(" を含むだけで落とす簡易検査のため、CSS としての妥当性や画面破壊までは制御していません。
また class/id を許すと、アプリ側 CSS や自前 JS が参照するクラス名と衝突する可能性があります。

起こり得る問題:
- Markdown 内 HTML が画面全体に重なる、透明化する、極端なサイズにするなどの表示妨害を起こす。
- mdv-copy-btn / mdv-table-ctrl / wrap など、アプリ内部用クラスとの衝突で編集・変換処理が壊れる。
- 将来 CSS や JS が増えた時に、許可済み class/id が攻撃面になる。

改善案:
- 通常表示では style を原則禁止し、必要な装飾だけ Markdown 構文や安全な class に変換する。
- ユーザー由来 class/id は prefix を付けるか禁止する。
- bleach など実績あるサニタイザの採用を検討する。
- サニタイズのユニットテストを追加する。


重大度: 中
5. SafeWebLoader の一時ファイルがウィンドウ終了まで増え続ける
該当箇所:
- main.py:992-1027 SafeWebLoader.load_html()
- main.py:1028-1030 cleanup()

内容:
1.9MB を超える HTML は tmpdir/page_N.html に連番で書き出されますが、古いファイルはウィンドウを閉じるまで削除されません。異常終了した場合は tmpdir 自体も残ります。

起こり得る問題:
- 巨大 Markdown を長時間編集すると /tmp 使用量が増え続ける。
- 異常終了後に mdvp_* ディレクトリが残る。
- 小さい HTML は setHtml、大きい HTML は file load になり、base URL や WebEngine の挙動差が出やすい。

改善案:
- 直近 N 件だけ残して古い page_N.html を削除する。
- 起動時に古い mdvp_* ディレクトリを掃除する。
- 可能なら常に同じロード方式に寄せ、base/security policy の差を小さくする。


重大度: 中
6. 設定保存とテーマ読み込みの失敗が見えない
該当箇所:
- main.py:368-383 load_settings()
- main.py:385-391 save_settings()
- main.py:491-510 load_plugin_themes()

内容:
設定やテーマの読み書きで例外を広く握りつぶしています。ユーザーには失敗が見えず、ログもありません。またテーマ値は必須キーの有無だけを見ており、CSS 色値として妥当か、危険な文字列が混じっていないかは検証していません。

起こり得る問題:
- 設定が保存されないのに正常に見える。
- 壊れたテーマファイルが無視され、原因がわからない。
- テーマ JSON の値が CSS 文字列として混入し、UI や生成 HTML を壊す。

改善案:
- ログファイル、ステータスメッセージ、ダイアログのどれかで失敗を通知する。
- 設定保存は一時ファイルへ書いて atomic rename する。
- テーマ値を #RRGGBB / rgba(...) など許可形式に限定する。


重大度: 中
7. 依存関係と実装フレームワークが一致していない
該当箇所:
- main.py:33-48 PySide6 を import
- MDViewerPro.spec:23-24 PySide6 を collect
- requirements.txt:103-107 PyQt6 系を pin
- README.md:3 PySide6 と説明

内容:
アプリ本体と PyInstaller spec は PySide6 前提ですが、requirements.txt には PyQt6 / PyQt6-WebEngine が pin されています。venv と venv_new も混在しており、どの環境が正なのか読み手が判断しにくい状態です。

起こり得る問題:
- 新規環境で requirements.txt を信じて入れると main.py が動かない。
- PyQt6 と PySide6 が同居し、Qt バージョン差やプラグイン競合でビルドが不安定になる。
- ライセンス・NOTICE・リリース説明と実配布物の整合性が崩れる。

改善案:
- requirements.txt を実行環境用に整理し、PySide6 / PySide6-Addons / PySide6-Essentials など実際の依存へ統一する。
- ビルド専用依存は requirements-build.txt や pyproject の optional group に分ける。
- venv / venv_new はリポジトリ管理外にし、再現手順を README に書く。


重大度: 中
8. ビルド成果物・仮想環境・配布物が作業ツリーに混在している
該当箇所:
- ルートに venv/, venv_new/, build/, dist/, MDViewerPro-Installer.dmg が存在
- git status で dist/ 配下の大量削除・変更、DMG 変更、venv_new 未追跡を確認
- .gitignore:10 に先頭スペース付きの " venv/" があり、意図通り無視されない可能性がある

内容:
ソースコード、仮想環境、PyInstaller 成果物、DMG、ビルドログが同じツリーに置かれています。レビュー時に本当のコード変更が生成物の差分に埋もれます。

起こり得る問題:
- 重要なコード変更や危険な差分を見落とす。
- 巨大バイナリを誤ってコミット・配布する。
- ローカル環境依存のファイルがリリース物に混ざる。

改善案:
- .gitignore を修正し、venv/, venv_new/, build/, dist/, *.dmg, __pycache__/, .DS_Store を確実に除外する。
- 配布物は GitHub Releases 等へ置き、Git にはソースと設定だけを置く。
- 生成物は必要なら別ディレクトリや CI artifacts で管理する。


重大度: 中
9. ビルドスクリプトがカレントディレクトリ依存で、削除範囲が荒い
該当箇所:
- build_dmg.sh:8 PYTHON="$(dirname "$0")/venv_new/bin/python3"
- build_dmg.sh:23 rm -rf build dist __pycache__ "${DMG_NAME}.dmg" dmg_tmp
- build_dmg.sh:26 PyInstaller MDViewerPro.spec
- fix_webengine.sh:5 APP="dist/MDViewerPro.app"

内容:
スクリプト自身の場所を使う箇所と、カレントディレクトリ相対で削除・ビルドする箇所が混在しています。別ディレクトリから実行した場合、意図しない場所の build/dist/dmg_tmp を削除したり、成果物が想定外の場所に出たりします。

起こり得る問題:
- 作業ディレクトリを間違えると別プロジェクトの build/dist を消す。
- fix_webengine.sh が期待する dist/ と PyInstaller の出力場所がずれる。
- CI や他の開発者環境で再現しにくい。

改善案:
- 冒頭で SCRIPT_DIR を絶対パス化し、cd "$SCRIPT_DIR" してから処理する。
- rm 対象は "$SCRIPT_DIR/build" のように絶対パスで限定する。
- set -euo pipefail を使い、失敗を握りつぶす箇所を減らす。


重大度: 中
10. コード署名・entitlements の扱いが配布設計として弱い
該当箇所:
- MDViewerPro.spec:162-163 codesign_identity=None / entitlements_file=None
- MDViewerPro.spec:191-194 Info.plist に entitlements 風キーを記載
- fix_webengine.sh:44-59 ad-hoc 署名、失敗を || true で無視
- README.md: 初回起動は未署名前提

内容:
entitlements は通常 Info.plist ではなく codesign の entitlements file で付与します。現状は ad-hoc 署名で、署名失敗も無視されます。未署名配布自体は可能ですが、QtWebEngine を含む macOS アプリとしては配布時の信頼性・診断性が弱いです。

起こり得る問題:
- Gatekeeper や macOS バージョン差で起動失敗する。
- 署名失敗してもビルド成功に見え、壊れた DMG を配る。
- allow-unsigned-executable-memory / disable-library-validation 相当の扱いが曖昧になる。

改善案:
- 開発用 ad-hoc 署名とリリース用 Developer ID 署名を分ける。
- entitlements.plist を用意し、codesign に明示的に渡す。
- codesign / spctl / notarization の検証をビルド手順に入れる。


重大度: 低-中
11. PDF書き出しがプレビュー WebView を一時的に差し替える設計
該当箇所:
- main.py:2766-2770 PDF用HTMLと元HTMLの生成
- main.py:2781-2785 _restore_original()
- main.py:2832-2835 loadFinished 接続後に PDF用HTML をロード

内容:
PDF 書き出しでは現在表示中の QWebEngineView に PDF 用 HTML をロードし、印刷後に元 HTML を再ロードします。接続順やタイムアウトは改善されていますが、プレビュー状態を副作用として差し替える設計はまだ脆いです。

起こり得る問題:
- PDF 出力中にユーザーが操作すると状態が競合する。
- スクロール位置、編集カーソル、MD編集モードの未反映入力が失われる。
- 印刷用テーマ適用や画像埋め込みの失敗が通常プレビューへ波及する。

改善案:
- PDF 専用の非表示 QWebEnginePage / QWebEngineView を別に持つ。
- PDF 出力中は編集・モード切替・再出力ボタンを一時的に無効化する。
- 元表示復元時にスクロール位置・モード・未保存状態を明示的に復元する。


重大度: 低-中
12. クラウドアップロード機能名と実動作にギャップがある
該当箇所:
- main.py:2645-2659 _upload_gdrive() / _upload_onedrive()

内容:
「Google Drive へアップロード」「OneDrive へアップロード」というメニュー名ですが、実際はブラウザで各サービスを開き、ユーザーに手動アップロードを促すだけです。

起こり得る問題:
- ユーザーが自動アップロード済みと誤解する。
- 未保存ファイルや保存前の変更がクラウドに反映されたと誤解する。

改善案:
- メニュー名を「Google Drive を開く」「OneDrive を開く」に変更する。
- 本当にアップロードするなら OAuth / API / 成功失敗通知まで設計する。


対策済みと判断した旧リスク
--------------------------
- QtWebEngine sandbox の常時無効化:
  main.py:20-24 で MDVP_DISABLE_SANDBOX=1 の明示時だけ無効化する形になっており、常時無効化ではない。

- Markdown 生HTML / script の未サニタイズ:
  main.py:840-986 に標準ライブラリベースの allowlist サニタイザが追加され、script / iframe / on* 属性などは除去される。
  ただし上記 4 の通り、allowlist の広さと WebChannel 境界には残リスクがある。

- 外部画像取得の UI スレッド同期実行:
  main.py:602-633 の ImageFetchWorker と main.py:2483-2494 によりバックグラウンド化されている。
  また main.py:540-565 で private / loopback / link-local 等を拒否している。

- PDF loadFinished の接続順競合:
  main.py:2776-2835 で接続を load_html より前に行い、タイムアウトも追加されている。


優先修正順
----------
1. ファイル読み込みのサイズ上限・バイナリ判定を入れる。
2. requirements.txt / .gitignore / venv・dist・DMG の管理方針を整理する。
3. MD編集モードの保存前差分確認、または MD編集の制限・警告を入れる。
4. WebChannel をモード別に最小化し、clipboard bridge を閲覧モードから外す。
5. サニタイザの class/id/style 許可を見直し、テストを追加する。
6. PDF 書き出しを専用 WebEngine インスタンスへ分離する。
